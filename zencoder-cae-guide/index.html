このトピックでは、コンテキスト・アウェア・エンコーディング（CAE）を使用するためのガイドを提供します。

Zencoder より焦点を絞ったサイトを提供するためにドキュメントが移動しました。 新しい場所は https://zencoder.support.brightcove.com。 新しいサイトを使用して、ブックマークを更新してください！ これらの古いドキュメントはまもなく削除されます。

概要

コンテキスト・アウェア・エンコーディング（CAE）は、各ソース・ビデオを分析し、各コンテンツのカスタムビットレートラダー（レンディションのセット）をインテリジェントに構築します。 さらに、コンテキスト認識符号化は、コンテンツを見るために使用される配信ネットワークおよびデバイスに関連する制約を考慮に入れている。 それは、必要なレンディションの数とそれぞれの解像度とビットレートを決定しますが、すべてのタイトルで一貫した品質を維持します。 これにより、ストレージと帯域幅のコストが大幅に節約され、ユーザーの再生エクスペリエンスが向上します。 追加情報については、 Context Aware Encoding の概要.

コンテキスト認識エンコーディングはすべてで利用可能ではありません Zencoder 顧客。 コンテキストアウェアエンコードを使用できるかどうかについては、アカウントマネージャーにお問い合わせください。

プロセス

CAEは、 Zencoder ジョブを設定して generate_dynamic_profile 〜へ true と追加 dynamic_profile_rendition 値をジョブの出力に設定します。 また、 dynamic_profile_optionsこれは必須ではありません。 ジョブの入力ソースファイルをダウンロードして検査した後、 Zencoder CAE動的プロファイルジェネレータをソースファイル上で実行します。 生成されたダイナミックプロファイルラダー設定は、エンコードされる前に、対応する各出力に追加されます。 dynamic_profile_renditionの値を持つOutputsには、次の設定が自動的に設定されます。

  • width
  • height
  • frame_rate
  • video_codec
  • video_codec_profile
  • video_codec_tier
  • video_codec_level
  • video_reference_frames
  • video_bframes
  • video_bitrate
  • decoder_bitrate_cap
  • decoder_buffer_size
  • keyframe_rate
  • fixed_keyframe_interval

無しのその他のアウトプット dynamic_profile_rendition ダイナミックプロファイルジェネレータが実行されている間に、入力ソースが検査されるとすぐにエンコードが開始されます。

設定のガイドライン dynamic_profile_options

レンディション数

作成するレンディションの最小数と最大数は、 min_renditions や max_renditions 設定。 最高のパフォーマンスを得るには、これらの制限値の間に一定のギャップを置いておくことをお勧めします（例：set min_renditions 2に max_renditions 10へ）、CAEプロファイルジェネレータが各コンテンツに最も適した番号を選択できるようにする。 コンテンツを簡単にエンコードするために、レンディションは少なくなる可能性がありますが、複雑なコンテンツの場合は、より多くのレンディションが生成されます。 より多くのレンディションが不要なときに帯域幅を節約できるように、min_renditionsを可能な限り小さく設定することもお勧めします。

決議

デフォルトでは、CAEは約30の標準解像度のラダーを使用します。 192x108 〜へ 7680x4320。 ローエンドまたはハイエンドのどちらでも制限することができます。 min_resolution や max_resolution 設定。 また、解像度と関連するコーデック設定のカスタムリストを、 video_configurations 設定。

GOPの長さ

挽き目 keyframe_rate 設定は最大値を制御します GOP エンコードされたコンテンツの長さ。 これに対応するように設定してください segment_seconds HLS / DASHエンコーディング用。 そうでなければ、0.5の値が推奨されます.2秒に対応します GOP 長さ

ビットレートカバレッジ

使用されるビットレートの限界は、 min_bitrate や max_bitrate 設定。 これはCAEに少なくとも min_bitrate 最初の演奏ではKbps、最後の演奏では最大でも max_bitrate Kbps。

レンディションのビットレートが適切に設定されていることを確認するには、 min_granularity や max_granularity 設定。 たとえば、あるビットレートから次のビットレートまでのステップが、ビットレートの50％増加以上、またはビットレートの100％増加以上であることを確認するために使用できます。 ほとんどのストリーミングオーサリングガイドラインでは、粒度は100％を超えないことが推奨されます。

スタートアップの待ち時間、不十分なネットワーク状態での信頼性

ビットレートラダーが常に低いビットレートで始まるようにするには、 max_first_rendition_bitrate 設定します。 最初の演奏ビットレートが低く設定されるほど、接続速度が遅いためにビデオプレーヤーがバッファリングする可能性は低くなります。 さらに、ストリーミングマニフェストが最も低いビットレートを最初にロードするように指示している場合、これも起動待ち時間に直接影響します。

レガシーデバイスのサポート

唯一可能な古いデバイスをサポートする Baseline プロファイルH.264を再生するには、 select_baseline_profile_configuration 設定します。 これが設定されていると、少なくとも1つのレンディションが Baseline プロフィール。 追加の制約（コーデックプロファイル、レベル、参照フレーム数、bフレームなど）は、カスタム video_configurations 設定。

注意してください select_baseline_profile_configuration is true デフォルトでは 少なくとも1つのレンディション 512x288以下の解像度を持ちます。

VBR可変制御

最大ピークビットレートは、 bitrate_cap_to_bitrate_ratio や bitrate_cap_offset 設定。 ビットレート "R"がレンディションに対して選択された場合、 デコーダ_bitrate_cap 次のように計算されます。

  decoder_bitrate_cap = R * bitrate_cap_to_bitrate_ratio + bitrate_cap_offset

同様に、デコーダのバッファ限界は、 buffer_size_to_bitrate_ratio や buffer_size_offset 設定。 ザ decoder_buffer_size 次のように計算されます。

  decoder_buffer_size = R * buffer_size_to_bitrate_ratio + buffer_size_offset

ピークビットレートおよびバッファ制限は、コーデックプロファイルおよび/またはレベル設定によっても制限されることに注意してください。 いくつかの一般的なデバイス互換性の問題は、 bitrate_cap_to_bitrate_ratio 2.0以下、および buffer_size_to_bitrate_ratio 〜へ 2.0 * GOP length （すぐに）。
