Zencoder：MP4とHLS出力を一緒に作成する

製品（複数可）
Zencoder
役割
API開発者
API
Zencoder API

このトピックでは、 透過的これは、あなたのビデオをMP4にエンコードし、それらをHLS出力として再パッケージするプロセスです。

概要

ライブストリームイベント開始時間 Zencoder ビデオをMP4にエンコードしてから、HLS出力として再パッケージ化することもできます（トランスコードではなく、「トランスミキシング」と呼ばれるプロセスを通じて）。 Transmuxingは既存のMP4ビデオをHTTPに必要なMPEG TSセグメントに再パッケージする Live ストリーミング（HLS）、ビデオファイルを再エンコードする必要はありません。 従属出力を使用すると、H.264ファイルとセグメント化されたファイルをすべて1つのジョブで作成することができます。 トランスミックスされた出力は、符号化コストの1 / 4で課金されます。 結果として得られるジョブは、6の出力を生成します.2はアカウントの通常のレートで、3はエンコードコストの1 / 4で、1はフリーです。

このために、2つの主要な出力セット（プレイリストを含む）を持つ単一のジョブを作成します。 最初の出力セットは、通常のH.264 / AAC MP4ファイルで、HLS出力のソースファイルとして動作するための特別な設定がいくつかあります。 2番目の出力セットは、それらのMP4ファイルを「ソース」入力として使用し、その内容をHLS出力に変換します。 これらは従属出力であるため、対応するソースが処理を完了するまで待ってから処理をスケジュールします。

このプロセスの重要な部分は、 sourceV2の新しいオプションである Zencoder API。 Sourceは、入力ファイルではなく、処理のためにジョブ上の別の出力によって作成されたファイルを使用するように出力に指示します。 この状況では、 Zencoder 要求されたビットレートで入力ファイルに基づいてH.264ファイルを作成します。 これらの出力のそれぞれが終了すると、セグメント化されたバージョンが出力から作成されます。

注：再生を円滑にする方法はありません transmuxとトランスコード出力を1つのプレイリストに結合する。 レンディションは同じ方法でエンコードする必要があります。

H.264出力

高ビットレートと低ビットレートを対象とした2 H.264ファイルを作成します。

{
    "label": "low",
    "format": "mp4",
    "video_bitrate": 200,
    "decoder_bitrate_cap": 300,
    "decoder_buffer_size": 1200,
    "audio_sample_rate": 44100,
    "height": "288",
    "url": "s3://example-bucket/low.mp4",
    "h264_reference_frames": 1,
    "forced_keyframe_rate": "0.1",
    "audio_bitrate": 56,
    "decimate": 2
},
{
    "label": "high",
    "format": "mp4",
    "video_bitrate": 1000,
    "decoder_bitrate_cap": 1500,
    "decoder_buffer_size": 6000,
    "audio_sample_rate": 44100,
    "height": "432",
    "url": "s3://example-bucket/high.mp4",
    "h264_reference_frames": "auto",
    "h264_profile": "main",
    "forced_keyframe_rate": "0.1",
    "audio_bitrate": 56
}

これらの出力から得られたファイルは、さまざまなデバイスで再生することができます。 それぞれが異なるビットレートと解像度をターゲットにしているため、ユーザーは適切なファイルを送信できます。 それぞれは、HTTPのセグメント化にも適しています Live ストリーミングと適応ビットレートストリームとしての機能

上記のリクエストにいくつかのオプションがあります：

  • forced_keyframe_rate 0.1に これは、ビデオが強制的に10秒ごとにキーフレームを持つようにします。 分割されたファイルの長さは10秒になります。これにより、各セグメントがキーフレームで始まることを保証します。
  • decoder_bitrate_cap ファイルの目標ビットレートを1.5xに設定します。 decoder_buffer_size ファイルの目標ビットレートを3.5xから5xに設定します。 これらの設定は、ファイル全体で一貫したビットレートを維持するのに役立ちます。その結果、セグメント化されたセグメントのサイズやビットレートがあまり変わらないようになります。

H.264ファイルが作成されたので、HTTPを作成する要求に追加の出力を追加できます Live 追加のエンコーディングを必要とせずに、ソースからセグメント化されたファイルをストリーミングします。

セグメント化された出力

上記のH.264ビデオおよびオーディオ出力のそれぞれは、 labels そのAPIオプションで これらのラベルは、 source 伝えるオプション Zencoder 入力ファイルではなく、指定されたラベルで出力によって作成されたビデオを使用します。 H.264ファイルは既にHLSの設定が良好であるため、追加のエンコーディングは必要ありません。

{
    "source": "low",
    "format": "ts",
    "copy_audio": "true",
    "copy_video": "true",
    "url": "s3://example-bucket/hls-low/hls-low.m3u8",
    "label": "hls-low",
    "type": "segmented"
}
{
    "source": "high",
    "format": "ts",
    "copy_audio": "true",
    "copy_video": "true",
    "url": "s3://example-bucket/hls-high/hls-high.m3u8",
    "label": "hls-high",
    "type": "segmented"
}
{
    "streams": [
    {
        "path": "hls-low/hls-low.m3u8",
        "bandwidth": 256
    },
    {
        "path": "hls-high/hls-high.m3u8",
        "bandwidth": 1056
    }
    ],
    "type": "playlist",
    "url": "s3://example-bucket/playlist.m3u8"
}

これらの出力から得られるファイルは、異なる帯域幅をターゲットとするセグメント化された出力と、再生デバイスが利用可能なストリームを知るための再生リストファイルである。

セグメント化された各出力には、 source前述の出力の1つを指定します。 彼らはまた、 copy_audio や copy_video オプション Zencoder ソースファイルからのオーディオとビデオのストリームを使用し、それらをトランスコードするのではなく、新しいコンテナで再パッケージします。 最後に、出力はまた、 type セグメント化された format のts、プラスa url .m3u8という拡張子を付けて Zencoder 分割されたHLSファイルを作成することを知っています。

最後に、セグメント化された各出力を参照する再生リストも生成されます。 プレイリストで参照される最初のファイルは低ビットレートビデオであることに注意してください。 ビデオを再生するデバイスは、通常、プレイリストに最初にリストされた最初のストリームをロードし、必要に応じて他のストリームに切り替えます。 これは、再生の最初の10秒がこのストリームを使用することを意味します。その後、再生する適切なストリームを選択するのに十分な情報がデバイスに格納されます。

注意事項

  • HLSプレイリスト生成は現在、ジョブの他の出力からの情報は使用しませんが、正しくフォーマットされた適応ビットレートプレイリストを簡単に生成し、それを他のファイルとともにアップロードする簡単な方法です。
  • iOSデバイスは適応ビットレートプレイリストを再生しますが、VLCは相対URLを処理する際のエラーのために作成されません。 これは、マニフェストファイルのディレクトリではなく、プレイリストファイルと同じディレクトリにあるセグメントファイルを検索します。
